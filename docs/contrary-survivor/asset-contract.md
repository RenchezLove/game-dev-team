## 1. Скелет

- **Имя объекта-арматуры:** `RootAnim` (тип ARMATURE). Датаблок арматуры внутри файла называется `Armature`. **[ЖИВ]**
- **Число костей: 21.** **[ЖИВ]**
- **Полный список костей** (порядок как в датаблоке) **[ЖИВ]**:
  `C_Root, C_Spine01, L_UpperArm, L_Shoulder, L_Arm, L_Hand, R_UpperArm, R_Arm,`
  `Pelvis.009_R.002, R_Hand, C_Spine02, C_Neck, C_Head, L_Pelvis, L_Thigh, L_Claf,`
  `L_Foot, R_Pelvis, R_Thigh, R_Claf, R_Foot`
- **Конвенция имён:** префиксы `C_` (center/осевые), `L_`/`R_` (left/right). Опечатка `Claf`
  (вместо Calf) — **намеренно сохраняется**, это часть существующего скелета.
- **Аномальная кость `Pelvis.009_R.002`** — занимает слот правого плеча/руки в цепочке
  (между `R_UpperArm` и `R_Hand`, фактически аналог `R_Shoulder`). Имя нестандартное, но
  **НЕ переименовывать** — на неё завязаны веса, анимации и UE-скелет (ADR-018, общий скелет
  для всех гуманоидов). **[ЖИВ имя + DOCX/ADR назначение]**
- **ЧТО НЕ ПЕРЕИМЕНОВЫВАТЬ:** ни одну кость, включая `Pelvis.009_R.002` и `*Claf*`. Любой
  новый гуманоид/модуль шарит **этот же** `RootAnim` (21 кость), иначе ломается Leader/Master
  Pose в UE и общие анимации.
- Бюджет костей по GDD: ориентир ~≤50–70 костей на персонажа; текущий скелет (21) с большим
  запасом. **[GDD]**

## 2. Масштаб / единицы

- **Единицы сцены Blender:** система METRIC, `scale_length = 1.0`, length unit = METERS →
  **1 Blender-единица (BU) = 1 метр.** **[ЖИВ]**
- **Рост гуманоида:** bbox по Z = **1.818 BU ≈ 1.82 м** (top головы Z≈1.819, низ стоп Z≈0.001). **[ЖИВ]**
- **Целевой масштаб в UE:** UE работает в сантиметрах, **1 UE-unit = 1 см**. FBX-экспортёр Blender
  пишет юниты как cm (Model-узлы получают scale 100 — проверено парсером FBX, раунд 3), поэтому
  **1 BU → 100 см** при импорте без доп. масштабирования. Рост в UE ≈ **182 UE-units (1.82 м)**. **[DOCX/контекст]**
- При импорте в UE **не задавать Import Uniform Scale ≠ 1.0** — иначе масштаб уедет.

## 3. Ориентация / оси экспорта

> Это проверенный рабочий пайплайн. **НЕ менять Forward/Up** — иначе сломается ориентация
> относительно меша игрока (тот импортирован тем же пайплайном).

- **Геометрия в Blender:** вертикальный гуманоид, Z-up, ground = Z≈0. **[ЖИВ]**
- **Результирующие GlobalSettings в FBX** (проверено парсером `io_scene_fbx.parse_fbx`, раунд 3,
  байт-в-байт совпали с рабочим эталоном `SK_BanditTorsoArmor3.fbx`) **[DOCX/контекст]**:
  - `UpAxis = Y`, `FrontAxis = Z`, `CoordAxis = X` (все знаки +1), формат **FBX 7400**.
  - Model-узлы: `RootAnim` Lcl Rotation `[-90,0,0]`, scale 100; меш Lcl Rotation `[-90,0,0]`;
    `C_Root` (LimbNode) Trans `[0,-0.001,0.839]`, Rot `[+90,0,0]`. Это штатная конверсия
    Blender Z-up → FBX Y-up; даёт в UE вертикального персонажа.
- В операторе экспорта оси **вручную не переопределялись** (дефолт Blender FBX) — конечный
  результат фиксируется именно через GlobalSettings выше. **[ЖИВ-TODO: явные `axis_forward`/`axis_up`
  операторе при следующем экспорте записать дословно]**
- Известно из раунда 3: «флип 180°» в одном из баг-репортов был **UE-стороной** (размещение
  актора/трансформ компонента), а не экспортом. Экспорт корректен.

## 4. Пивоты / origin

- **Origin всех 3 мешей (Head/Torso/Legs): мировые (0,0,0).** **[ЖИВ]**
- **Origin/трансформ арматуры `RootAnim`:** location (0,0,0), rotation (0,0,0,°), scale ≈ (1,1,1)
  — rest-поза, без накопленных трансформов. **[ЖИВ]**
- **Ground = Z0:** низ стоп (`Legs` min Z ≈ 0.0012) практически на полу; персонаж стоит «на нуле»,
  растёт вверх по +Z. В UE это даёт корректную привязку капсулы к подошвам. **[ЖИВ]**
- **Перед экспортом базового меша — сброс в rest-позу** (Pose mode → Select All → Clear Transforms;
  удалить action). База master-файла приходит в RUN-позе (action `ArmatureAction`, frame 3) — это
  поза анимации, не bind; одежду/новую геометрию авторить на rest. **[DOCX/контекст]**

## 5. Модульность (ADR-018)

- **3 слота: `Head`, `Torso`, `Legs`** — отдельные скинённые меши, **общий скелет `RootAnim`**
  (21 кость). **[ЖИВ: 3 меша, parent=RootAnim, Armature-модификатор на каждом]**
- Z-зоны слотов (мировые, эталон-бандит) **[ЖИВ]**:
  - `Head`  Z ≈ [1.532 … 1.819]
  - `Torso` Z ≈ [0.904 … 1.549]
  - `Legs`  Z ≈ [0.001 … 0.905]
  - Зоны **нахлёстываются** по Z (Head↔Torso ~1.53–1.55; Torso↔Legs ~0.90) — это намеренный
    **нахлёст без щелей** на стыках, чтобы при деформации не появлялись дыры.
- **1 скинённый меш на слот.** Новые части одежды (кепка, оболочка бомбера, лампас и т.п.)
  строятся как добавленная геометрия и **джойнятся** в исходный слот-меш. **[DOCX/контекст]**
- **Веса новых вершин:** переносить с **ближайшей исходной вершины (KDTree)** → корректная
  деформация общим скелетом. Проверено позой: clipping/отрыва нет. **[DOCX/контекст]**
- В UE модули шарят скелет через **Leader/Master Pose Component**, одна анимация; для врагов/NPC
  предусмотрен **runtime mesh merge при спавне** (склейка в 1 скелетный меш → 1 draw call).
  Реализация merge на стороне cpp-dev/UE. **[GDD §2 / ADR-018]**

## 6. Vertex-color / материалы

- **1 материал на все слоты: `M_Bandit_Flat`** (Principled BSDF, Base Color из vertex-color). **[ЖИВ: материал на всех 3 мешах]**
- **Vertex-color атрибут: имя `Col`, домен `CORNER`, тип `BYTE_COLOR`.** Одинаково на всех 3 мешах. **[ЖИВ]**
- Параметры материала: Base Color = нода `ShaderNodeVertexColor` (layer `Col`), Metallic 0,
  Roughness 0.65, Specular 0.15, без normal-map. Vertex-color в линейном пространстве. **[контекст]**
- **Палитра (hex):** skin `C9A084`, jacket `2B2B2E`, tee `6E6E70`, joggers `303033`,
  sneaker `ECECEC`, cap `28282B`, rust `C64B2C`. **[контекст]**
- UV: один слой `UVMap`, Smart-UV в диапазоне 0–1 без overlap (модель готова под бейк). **[ЖИВ: UVMap есть; чистота 0–1/overlap — контекст]**
- **Известная проблема (раунд 5): границы цветовых зон «расплываются»/сдвигаются в UE.**
  На low-poly без рёбер вдоль границ зон domain CORNER/BYTE_COLOR при FBX→UE интерполируется
  per-vertex → визуальный сдвиг (низ голеней белеет выше щиколотки; rust-зона уезжает на предплечье).
  Color-space (sRGB↔linear) на стороне UE закрывается материалом (Power 2.2); остаётся **геометрия границ**.
- **Рекомендация для финал-пасса:** **бейк цветовых зон vertex-color → текстуру** (UV уже чистые 0–1):
  надёжнее на low-poly, снимает gamma-разночтения и проблему границ; альтернатива — разрез геометрии
  по границам зон (дороже по полигонам). Выбор финального решения — отдельная задача финал-арта. **[контекст]**

## UE-импорт (подтверждено unreal-operator, 2026-06-13)

> Прочитано вживую из `InterchangeAssetImportData` ассетов бандита (read-only, ничего не менялось).

- **Пайплайн импорта:** UE 5.5 **Interchange** (`InterchangeGenericAssetsPipeline`), НЕ legacy-FBX. Меши приняты как `IFMT_SKELETAL_MESH`.
- **Путь импорта:** `/Game/Characters/Bandit/` — 3 SkeletalMesh (`SK_Bandit_Head/Torso/Legs`) + материал `M_Bandit_VColor`. Игрока в `/Game/Characters/` пока нет (только бандит).
- **Источник FBX:** `E:/ForGameLead(Materials)/bandit-work/SK_Bandit_*.fbx` (вне репозитория, рабочая папка modeler-3d).
- **Import Transform (на пайплайне) = identity:** `import_offset_translation`=(0,0,0); `import_offset_rotation`=(0,0,0); `import_offset_uniform_scale`=**1.0**. Геометрия принимается как есть из FBX (никакого доворота/смещения/домасштаба со стороны UE).
- **Масштаб:** uniform scale 1.0; **1 UE-единица = 1 см** (движковая константа). Согласовано с Blender-стороной (1 BU = 1 м, FBX scale100) — стык корректен.
- **Skeleton (общий, ADR-018):** все 3 части ссылаются на `/Game/TestContentAndCode/PreProduction/HeadAndSkeletonfbx_Head_Skeleton` — единый скелет для модульной сборки/merge.
- **Vertex Color Import Option:** `IVCIO_REPLACE` (Replace), override-color (0,0,0,0) — vertex color берётся из FBX и заменяет существующий; согласуется с материалом `M_Bandit_VColor` (нода VertexColor → Power(2.2) → BaseColor).
- **Прочие флаги Interchange:** `import_meshes_in_bone_hierarchy`=True, `bake_meshes`=True, `recompute_normals`=True, `recompute_tangents`=True, `use_mikk_t_space`=True, `use_t0_as_ref_pose`=False.
- Без Create Physics Asset на частях (модульные меши).
- Нейминг в UE: `SK_` (SkeletalMesh), `M_` (Material), `MI_` (MaterialInstance) — придерживаться.
- Текстуры под Android — при переходе на бейк-текстуры (§6).
