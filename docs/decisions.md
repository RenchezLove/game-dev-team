# Архитектурные решения (ADR)

> Каждое решение по инструментам и архитектуре фиксируется здесь. «Нативное проверено» означает живую проверку реестра коннекторов, а не память.

## ADR-001 — Движок системы: субагенты Claude Code, standalone, game-lead = корень
Система game-dev — самостоятельная (не вложена в другие системы агентов). Движок — субагенты Claude Code (`.claude/agents/*.md`). `game-lead` запускается как главная сессия (`claude --agent game-lead`) и порождает специалистов.
**Причина:** субагент Claude Code не может порождать субагентов (подтверждено по code.claude.com/docs). Делая `game-lead` корнем-главной-сессией, получаем рабочее дерево «лид → специалисты». Будь game-lead вложенным субагентом, он не смог бы звать специалистов. Запас на многоуровневость в будущем — через agent teams (отдельный механизм), не через субагенты.

## ADR-002 — modeler-3d: Blender через официальный сервер blender-mcp (CLI-уровень)
Инструмент 3D — официальный сервер **blender-mcp v1.0.0** (Blender Lab), заведённый в Claude Code **CLI**, НЕ через бандл Claude Desktop.
- Установка: `uv tool install` из `git+https://projects.blender.org/lab/blender_mcp.git#subdirectory=mcp`. Бинарь: `C:\Users\pgr40\.local\bin\blender-mcp.exe`.
- Регистрация: проектно в `.mcp.json` (сервер `blender`).
- Привязка к агенту `modeler-3d` через `mcpServers: [blender]` (только этот агент видит инструменты Blender).
- Сторона Blender: аддон **Blender Lab v1.0.0**, слушает `localhost:9876`, auto-start.
Самодельный MCP на `bpy` не используем — есть официальный сервер.
**Статус:** заведено и проверено живыми вызовами (get_objects_summary, get_object_detail_summary, render_viewport_to_path). Рабочие нюансы — см. [ADR-011](#adr-011--blender-рабочие-нюансы-один-клиент-на-сокет-output_path-у-render_viewport).

## ADR-003 — unreal-operator: MCP-сервер @runreal/unreal-mcp (UE Python/Remote Execution под капотом)
Нативного коннектора Unreal в реестре Claude нет (проверено несколькими запросами — граница проверки), встроенного навыка тоже нет → по правилу «нативное первым» спускаемся на уровень MCP. Выбран сервер **`@runreal/unreal-mcp`**, который ходит в редактор по UE Python / Remote Execution. Computer Use остаётся крайним случаем, если программного пути не хватит.
- Регистрация: проектно в `.mcp.json`, сервер `unreal`. **Запуск через абсолютный путь к `node.exe` + абсолютный путь к `dist/bin.js` пакета** (`command: "C:\\Program Files\\nodejs\\node.exe"`, `args: ["…\\@runreal\\unreal-mcp\\dist\\bin.js"]`) — надёжная форма, по аналогии с blender (полный путь к бинарю).
  - **Почему не голый `npx`:** 2026-06-06 первая попытка упала на старте — `claude mcp list` показал `unreal … ✗ Failed to connect`, тулы `mcp__unreal__*` в сессию не загрузились. Причина: неинтерактивная оболочка, через которую харнесс стартует MCP-серверы, **не видит `C:\Program Files\nodejs` в PATH** (`npx`/`node` NOT FOUND), хотя Node реально установлен (node v24.16.0). blender-сервер цепляется, т.к. задан полным путём к `.exe`. Фикс — указать **абсолютный путь к `node.exe`** и прямо к `dist/bin.js` пакета, минуя резолв `npx`.
- **Multicast Bind Address = `0.0.0.0` обязателен** (UE → Project Settings → Plugins → Python Remote Execution). С дефолтным loopback-адресом multicast от node-клиента **не сходится** с редактором, тулы не отвечают. На Windows ставить именно `0.0.0.0`.
- **Редактор UE должен быть открыт и ПОЛНОСТЬЮ загружен ДО старта `claude`.** Если проект ещё грузится / редактор не поднят — MCP-сервер `unreal` стартует и сразу падает (`exit → ✗ Failed to connect`), тулы в сессию не попадают. Порядок: сначала открыть проект в UE и дождаться полной загрузки, затем запускать `claude`.
- Привязка к агенту: `unreal-operator` через `mcpServers: [unreal]` (только он видит `mcp__unreal__*`, по правилу из CLAUDE.md).
- Сторона UE: проект открыт в редакторе, **Python Editor Script Plugin** + **Enable Remote Execution** включены; к редактору подключается один клиент.
- **Предусловие рантайма:** серверу нужен **Node.js**. 2026-06-06 установлен `winget install OpenJS.NodeJS.LTS` → **node v24.16.0**, npm 11.13.0 (`C:\Program Files\nodejs`). Пакет `@runreal/unreal-mcp` существует (`npm view` → version 0.1.4).
**Статус: ПРОВАЛИДИРОВАН живыми вызовами** (2026-06-06, смоук-тест в сессии). Поднялось **20 тулзов `mcp__unreal__*`**. `editor_project_info` → проект **ContrarySurvivor**, движок **UE 5.5.4** (`5.5.4-40574608+++UE5+Release-5.5`), **GAS + Enhanced Input** включены, 7233 ассета. Запись/чтение/удаление проверены: create+delete `SmokeTestCube` (outliner 5→6→5), `HighResShot` дал реальный PNG (визуально подтверждён Ринатом).
- **Рабочие квирки (проверено живыми вызовами):**
  - **Скриншоты:** рабочий рецепт постоянного снимка — **окно UE в фокусе** → `editor_console_command HighResShot <WxH>` → файл в `<ProjectDir>/Saved/Screenshots/WindowsEditor/`. Без фокуса high-res пайплайн **не тикает** и не завершается (`is_task_done()` висит). `editor_take_screenshot` рендерит корректно, но пишет в **эфемерный tmp** (harness удаляет до след. вызова) — для отдаваемых артефактов непригоден. SceneCapture2D из Python даёт **неосвещённый/чёрный** кадр — не использовать для скринов сцены.
  - **`get_unreal_project_path()` в обёртке возвращает «not set»** → путь брать из `editor_project_info` / `unreal.Paths` (либо разово выставить `set_unreal_project_path`). Реальный путь проекта: `E:/ContrarySurvior/ContrarySurvivor/`.
  - **`editor_get_world_outliner` отдаёт внутренние имена актёров, не labels** → для точечных операций резолвить label через Python либо удалять/искать по label (`editor_delete_object` по label работает).

## ADR-004 — concept-artist: визуальный инструмент + сужение скоупа роли (ПЕРЕСМОТРЕН)
Нативного коннектора-генератора изображений в реестре Claude нет. Инструмент роли `concept-artist` — **нативный code-render**: визуал строится кодом (`HTML/CSS` + инлайн `SVG`) и рендерится в `PNG/PDF` через **headless Microsoft Edge (Chromium)**; письменная арт-дирекшн — в `.md`. Внешние AI-генераторы изображений не вводим.

**Сужение скоупа роли (2026-06-08, решение Рината — закрытие роли, Хвост 1).** Скоуп `concept-artist` сужен до того, что нативный тулсет закрывает с реальной практической пользой:
- **В скоупе:** (1) палитры / цвето- и типосистемы (HTML/SVG → PNG/PDF); (2) UI-макеты (HUD, меню) на HTML/CSS — сильная сторона роли; (3) письменная арт-дирекшн в `.md` (настроение, референс-разбор, обоснование палитры); (4) текстовые спек-листы на ассет для `modeler-3d` (ключевые формы, пропорции, размеры, лимит полигонов, референсы).
- **Убрано из скоупа:** силуэты/блокауты форм и градиент-мудборды — на практике пользы не давали. Не делать, не имитировать.
- **Честная граница (нативно не закрывается):** фигуративный/живописный концепт-арт (детальный рисунок персонажа/сцены «от руки») требует генератора изображений, которого в тулсете нет → `concept-artist` эскалирует `game-lead`, не имитирует code-render'ом.

**Генерацию изображений и Claude Design рассмотрели и НЕ вводим (2026-06-08):**
- **Нативная генерация картинок Claude — нет.** Claude не генерирует изображения нативно; коннектора-генератора в реестре Claude нет (граница проверки — реестр коннекторов). Поэтому фигуративный арт остаётся вне тулсета (см. границу выше).
- **Claude Design (он же «canvas»/web-инструмент дизайна) — не подключается к Claude Code как тул.** Это отдельный продукт/поверхность, не коннектор и не навык, доступный из Claude Code; подключить его как инструмент роли нельзя. Не вводим.
- **Внешние AI-генераторы изображений — против native-first и духа ADR-004.** Сторонние генераторы (вне нативного коннектора/навыка) нарушают правило «нативное первым» из CLAUDE.md и предыдущее решение ADR-004 (внешние генераторы не вводим). Не вводим.
- Итог: фигуративный концепт-арт сейчас закрытого нативного пути не имеет — это осознанная граница роли, а не временная заглушка.

**История решения (исходное допущение про canvas-design не подтвердилось):**
- **(a)** Первая редакция ADR-004 предполагала «нативный навык canvas-design (png/pdf)» со статусом «проводка на Фазе инструментов». Живая валидация (Ш7, 2026-06-08) показала: навык **не был установлен и недоступен** — ни как preload, ни on-demand. Проверено тремя независимыми наблюдениями со стороны `concept-artist` (teammate) + подтверждено `game-lead` (главная сессия): в system-reminder реестре навыков `canvas-design` отсутствует у обоих; прямой `Skill(canvas-design)` → `Unknown skill`; ФС-скан `.claude/skills/**` пуст. Это **не** баг прокидывания в teammate (#29441) — навыка не было в установке вообще.
- **(b)** По содержанию (сверено по `anthropics/skills`, решение Рината 2026-06-08) `canvas-design` — навык **абстрактного искусства / постеров**, а НЕ под функциональный скоуп роли (палитры, мудборды, силуэты/блокауты, UI-макеты, арт-дирекшн). Поэтому основным инструментом роли его не делаем, даже будучи установленным.
- **(c)** Решение: основной инструмент — **нативный code-render через Edge headless**. Рабочая форма подтверждена живым рендером (2026-06-08): `msedge.exe --headless=new --disable-gpu --hide-scrollbars --no-sandbox --window-size=W,H --screenshot=OUT.png "file:///ABS.html"` → реальный валидный PNG. PDF — тем же Edge с `--print-to-pdf`. Toolchain машины: Edge есть; Python/Pillow НЕТ (только Store-заглушка) → палитры/силуэты идут через SVG/HTML→Edge; Node v24 есть (puppeteer/sharp — запас). Инструменту `concept-artist` добавлен `Bash` (без него нативный путь неисполним).
- **(d)** **Ограничение (нативно не закрывается):** фигуративные/живописные концепт-зарисовки (детальный рисунок персонажа/сцены «от руки») требуют генератора изображений, которого в нативном тулсете нет. Code-render технически способен и на силуэты/блокауты/мудборды, но из практического скоупа роли они исключены (см. блок «Сужение скоупа роли» выше) — актуальный скоуп: палитры/типосистемы, UI-макеты, арт-дирекшн, текстовые спек-листы. Фигуративный рисунок code-render не закрывает в принципе. Такие задачи `concept-artist` эскалирует `game-lead`, не имитирует.

**Статус: ПЕРЕСМОТРЕН — нативный code-render (Edge headless), валидирован живым рендером Ш7 (2026-06-08); скоуп роли сужен и закрыт (2026-06-08, Хвост 1).** См. также инструкцию роли `.claude/agents/concept-artist.md`.

## ADR-012 — unreal-operator: автономный скриншот без фокуса окна (take_high_res_screenshot)
Рабочий рецепт **автономного** снимка редактора UE 5.5 **без вывода окна в фокус и без кликов** — для отдаваемых артефактов. Уточняет/дополняет квирк «Скриншоты» из [ADR-003](#adr-003--unreal-operator-mcp-сервер-runrealunreal-mcp-ue-pythonremote-execution-под-капотом): рецепт с `HighResShot`+фокус остаётся валидным для ручного снимка, ниже — автономная альтернатива.

**Рецепт (проверен живыми вызовами 2026-06-07):**
1. Через `editor_run_python` вызвать `unreal.AutomationLibrary.take_high_res_screenshot(W, H, out)`, где `out` — **постоянный** путь (напр. `<ProjectSavedDir>/Screenshots/WindowsEditor/auto_shot.png`, папку создать `os.makedirs(..., exist_ok=True)`).
2. Готовность проверять **по факту записи файла**: `os.path.exists(p)` + `os.path.getsize(p)` через `editor_run_python`. Файл пишется отложенно (по завершении кадра), обычно ~3–5 с; при NOT_YET — подождать и перепроверить.
3. **НЕ использовать `editor_take_screenshot`** — рендерит в эфемерный tmp, который harness удаляет до следующего вызова (для отдаваемого артефакта непригоден).
4. **НЕ использовать `HighResShot`** для автономного режима — high-res пайплайн требует фокуса окна, без фокуса не тикает и `is_task_done()` висит.

**Обязательное предусловие (иначе файл не запишется):**
- Настройка редактора **"Use Less CPU when in Background" = OFF** (троттлинг CPU в фоне выключен).
- Окно редактора **открыто и НЕ свёрнуто** (свёрнутое/троттлящее окно не тикает кадры → `take_high_res_screenshot` ставит запрос в очередь, но PNG не пишется).

**Доказательство (протокол B):** 1-й прогон 2026-06-07 при включённом троттлинге → `NOT_YET` (файл не появился за ~6 с). После OFF + развёрнутого окна — повтор тех же шагов → `OK`, `E:/ContrarySurvior/ContrarySurvivor/Saved/Screenshots/WindowsEditor/auto_shot.png`, **1 594 187 байт** (подтверждён `os.path.getsize`).

**Ограничение: предусловие машинное, не проектное — в репо не попадает.**
- "Use Less CPU when in Background" — это ключ `bThrottleCPUWhenNotForeground` в секции `[/Script/UnrealEd.EditorPerformanceSettings]` (класс `UEditorPerformanceSettings`, модуль `UnrealEd`).
- Хранится в **per-user/машинном** файле `%LOCALAPPDATA%\UnrealEngine\5.5\Saved\Config\WindowsEditor\EditorSettings.ini` (фактически найдено: `C:\Users\pgr40\AppData\Local\UnrealEngine\5.5\Saved\Config\WindowsEditor\EditorSettings.ini`, строка `bThrottleCPUWhenNotForeground=False`).
- В конфигах **проекта** (`E:/ContrarySurvior/ContrarySurvivor/Saved/Config/**`, `.../Config/**`) ключа **нет** (рекурсивный поиск — пусто). Значит **в систему контроля версий не попадает** и на другой машине/у другого пользователя его нужно **выставлять заново вручную** (Editor Preferences → General → Performance → снять «Use Less CPU when in Background»).
- Гипотеза для переноса в репо (НЕ проверено, требует дизайн-решения): значение потенциально можно зафиксировать на уровне проекта через `Config/DefaultEditorPerProjectUserSettings.ini` с той же секцией/ключом — такого файла в репо сейчас нет. Применять только после живой проверки, что project-default перебивает per-user значение.

## ADR-013 — ContrarySurvivor: риск дрейфа AFS SecurityToken в публичном репо (ЗАКРЫТО — риск принят)
Контекст: проектные `Config/*.ini` репозитория **ContrarySurvivor** (`github.com/RenchezLove/ContrarySurvivor`, **публичный**) заведены под VCS, чтобы зафиксировать настройки runreal MCP (`bRemoteExecution=True`, `RemoteExecutionMulticastBindAddress=0.0.0.0`). При этом в `Config/DefaultEngine.ini`, секция `[/Script/AndroidFileServerEditor.AndroidFileServerRuntimeSettings]`, лежал dev-токен `SecurityToken` (плагин **Android File Server**, AFS). Перед первым трекингом значение обнулено (`SecurityToken=`) — в git-историю реальный токен **не попадал** (коммит `f7542ad`); сквозной скан остальных `Config/*.ini` секретов не нашёл.

**Решение (правило):** секрет в репозиторий не кладём — независимо от публичности репо. Это не разовая мера, а постоянное правило для проектных конфигов под VCS.

**Остаточный риск (контингентный):** файл `Config/DefaultEngine.ini` теперь трекается. Если AFS-плагин активно используется, UE может **локально перегенерировать** `SecurityToken` → появится непустое значение, которое легко случайно закоммитить и запушить в публичный репо.

**Рассмотренные варианты защиты (ни один НЕ внедряется):**
- **(a) `git update-index --skip-worktree Config/DefaultEngine.ini` — ОТКЛОНЁН.** Обоснование: флаг локальный, **не переезжает на другую машину** (на ноуте защиты не будет), и **прячет легитимные правки** настроек этого файла (риск незаметно потерять/проглядеть нужное изменение конфигурации).
- **(b) Вынести AFS-секцию/токен в per-user конфиг вне VCS** — НЕ внедряется (см. статус).
- **(c) Отключить плагин AFS** — НЕ внедряется (AFS используется, см. статус).

**Статус: ЗАКРЫТО — риск принят, защита не вводится** (решение Рината, 2026-06-07). Спец-меры (a)/(b)/(c) не внедряем; `Config/DefaultEngine.ini` остаётся под VCS как есть, изменений в репозитории игры по этому поводу не делаем.
**Причина принятия риска:** Ринат использует WiFi-заливку на телефон через Android File Server (плагин нужен, отключать нельзя), а `SecurityToken` — **dev-only** (`bIncludeInShipping=False`) и работает **только в локальной сети**; ценность токена низкая, потенциальный ущерб от его раскрытия незначителен. Поэтому контингентный риск дрейфа токена в публичный репо принят осознанно.
**Действующее предупреждение (остаётся в силе):** репозиторий `Config/` **публичный** — настоящие секреты (ключи API, ключи подписи/keystore, облачные креденшелы, пароли) туда класть НЕЛЬЗЯ. Принятие риска касается исключительно низкоценного dev-токена AFS, а не секретов вообще.

## ADR-014 — protect-claude: PreToolUse-сторож рельсов от субагентов (best-effort, не airtight)
Контекст (Ш9 T2): включаем «тихий» режим (bare `Bash`/`PowerShell` в `permissions.allow`, без промптов) — человек выходит из петли подтверждений. Чтобы напарник/субагент не мог переписать собственные рельсы команды (хуки, settings, agent-дефы, git-хуки), добавлен PreToolUse-хук `.claude/hooks/protect-claude.ps1`.

**Механика.** Детект субагента — по полю `agent_id` в payload хука: оно присутствует **только** для вызовов субагента/напарника; главная сессия `claude --agent game-lead` его НЕ имеет (источник: code.claude.com/docs/en/hooks.md, 2026-06-09). Нет `agent_id` → exit 0 (пропуск). Есть `agent_id` + запись в защищённое дерево → exit 2 (block).
- **Защищённое множество:** всё под `.claude/**` и `.githooks/**` (суперсет явного списка из постановки — чтобы любой брошенный в `.claude` файл тоже резался). Карвут: `.claude/QA_OK` (маркер мерж-гейта) — точный финальный сегмент, traversal `QA_OK/../hooks` и суффикс `QA_OK_evil.txt` НЕ проходят.
- **Покрытие:** Write/Edit (по `file_path`) + Bash/PowerShell-команды (редиректы `>`/`>>` в защищённый путь, write-глаголы: Set-Content/Add-Content/Out-File/Tee-Object/New-Item/Remove-Item/Move-Item/Copy-Item/sed -i/cp/mv/rm + .NET WriteAllText/AppendAllText/StreamWriter/FileStream/::Create/::OpenWrite + tee/dd/truncate). Read-only (cat/Get-Content/git show) — пропускаются.
- Регистрация в `settings.json` `PreToolUse`, matcher `Edit|Write|Bash|PowerShell`. Настройки кешируются на старте сессии → **активируется только после рестарта**.

**Граница честности (важно — НЕ airtight).** Детект записи — это **denylist глаголов**, он принципиально неполон. Подтверждённые остаточные классы (qa, 2026-06-11):
- **Under-block (дыра):** inline-запись через интерпретатор — `python -c "open('.claude/..','w')"`, node `fs.writeFileSync` — НЕ ловится без ложных срабатываний на легитимных чтениях.
- **Over-block (fail-safe):** read-команда, буквально содержащая слова `tee`/`dd`/`install`/`truncate`/`create` И ссылающаяся на защищённый путь, → ложный block. Безопасная сторона (никогда не «ложно пропустить»), низкое влияние.

**Почему этого достаточно как слоя.** protect-claude — **defense-in-depth**, не единственная защита. Целостность `master` держится **независимо**: git-native pre-push/pre-merge хуки (T1) + маркер `.claude/QA_OK`, который создаёт **только** game-lead после «добро» qa. Напарники в `master` не пушат и не мёржат (ветка-политика + хуки), поэтому даже подмена рельсового файла в фиче-ветке не даёт напарнику протащить изменение в `master` — game-lead увидит diff до merge.

**Решение по остаткам:** НЕ чинить «точечным» анкорингом глаголов сейчас — это рискует ввести **under-block** (напр. `MSYS_NO_PATHCONV=1 rm .claude/...` при анкоринге по разделителям), что хуже fail-safe over-block. Оба остаточных класса вынесены в отдельный **follow-up** (вариант надёжного фикса — резолвить целевой путь записи / OS-level read-only ACL на `.claude/hooks` и `.githooks` для процесса агента, а не перечислять глаголы).

**Статус: ВНЕДРЕНО, прошло мерж-гейт qa** (ветка `feat/protect-claude-and-quiet`, коммит `13858c9`; qa ATTEST PASS с артефактом — харнесс 31/31, прогон против боевого хука). Активация — после рестарта сессии.

## ADR-015 — ContrarySurvivor: статы/выживание без GAS, на кастомном float-компоненте
**Контекст.** Диздок требует статы персонажа (Health, Hunger, Thirst, Money) и механику выживания (деградация голода/жажды, урон при истощении). UE-проект включает плагин GameplayAbilities (GAS), но в C++ он не используется — статы объявлены как «голые» float в классе персонажа, без системы.

**Решение (Ринат, 2026-06-11).** **GAS НЕ используем.** Статы и выживание реализуем через **лёгкий кастомный `UStatsComponent`** (`UActorComponent`) на обычных `float`:
- Поля: `Health/MaxHealth`, `Hunger/MaxHunger`, `Thirst/MaxThirst`, `Money`, `MoveSpeed`.
- Деградация Hunger/Thirst по таймеру (Tick/Timer), при пороге «критично» — деградация Health.
- Компонент **расширяемый под модификаторы статов** (структура модификаторов аддитив/мультипликатив) — чтобы при желании накинуть перки/баффы позже без переписывания. В MVP модификаторы используются только бронёй (защита) и снаряжением.
- Делегаты на изменение статов (`OnHealthChanged` и т.п.) для привязки HUD.

**Причина.** GAS даёт сетевую репликацию, GameplayEffects, теги — это избыточная сложность для одиночной (на MVP) игры с простыми статами и противоречит просьбе Рината «не усложнять». Кастомный компонент проще в отладке, дешевле по производительности (важно для Android), достаточен под текущий и обозримый скоуп. Прогрессия в игре — **через снаряжение, без XP** (см. ADR ниже по прогрессии в GDD ч.7.5), что снимает потребность в GAS-атрибут-сетах.

**Граница.** Если на пост-MVP появится мультиплеер или сложная система абилок — решение пересматривается (GAS может вернуться). Сейчас — кастом.

**Статус: ПРИНЯТО (Ринат, 2026-06-11).** Реализация — Фаза 1 (`UStatsComponent` базовый: Health + смерть; выживание Hunger/Thirst — Фаза 2). Источник правды по числам выживания — GDD ч.7.3 (черновые, утверждены на тюнинг).

## ADR-016 — ContrarySurvivor: World Partition отложен, старт на одной малой карте
**Контекст.** Диздок описывает большую основную локацию «Окрестности деревни» (~7-8 деревень по площади) + подлокации. UE 5.5 предлагает World Partition (стриминг большого открытого мира).

**Решение (Ринат, 2026-06-11).** **World Partition отложен.** Стартуем и ведём MVP на **одной малой карте**. Большой мир / стриминг вводим отдельной фазой построения мира (пост-MVP), когда контент и вертикальный срез готовы.
- Первый вертикальный срез (Фаза 1) — **маленький кусок «окрестностей деревни»** (решение Рината): простой уровень, 1-2 дома (пока кубы-заглушки), 1-2 дерева, 1-2 крупных камня, тест-арена под бой.

**Причина.** WP — это инфраструктура под масштаб, которого на MVP нет; раннее внедрение тормозит итерации и усложняет отладку. Малая карта достаточна для вертикального среза и ранних фаз. Переход на WP — обратимый рефакторинг уровней, не блокирует архитектуру геймплея.

**Статус: ПРИНЯТО (Ринат, 2026-06-11).**

## ADR-017 — ContrarySurvivor: абстракция ввода и наведения (ПК сейчас, тач Android позже)
**Контекст.** Цель — ПК + Android. Диздок: на ПК прицел/выбор цели мышью, на Android — виртуальные стики/кнопки и тап. Нужно не загнать архитектуру в мышь-специфику, чтобы потом не переписывать под тач.

**Решение (Ринат, 2026-06-11).**
- **Наведение через абстракцию «выбрать цель в точке экрана».** ПК: цель = актор под курсором мыши; клик ЛКМ = имитация тапа. Тот же путь («точка экрана → выбор цели/действие») переиспользуется для **тача Android** позже.
- **Первый срез — клик-захват цели** (выбрал кликом, цель держится при движении). Авто-захват ближайшего врага в радиусе ~5 м (диздок) — добавляем после первого среза.
- **Архитектуру ввода держим абстрактной** (Enhanced Input + слой «намерение действия», не привязанный к конкретному устройству). Реализация виртуального тач-UI Android — отдельная фаза.

**Причина.** Раннее разделение «устройство ввода ↔ игровое намерение» исключает дорогой рефактор при добавлении тача. Enhanced Input UE 5.5 уже даёт абстракцию маппинга; поверх — тонкий слой выбора цели по экранной точке, единый для мыши и тача.

**Статус: ПРИНЯТО (Ринат, 2026-06-11).** Реализация наведения/клик-захвата — Фаза 1.

## ADR-018 — ContrarySurvivor: модульность для ВСЕХ персонажей + runtime mesh merge как рычаг стоимости
**Контекст.** В Фазе 0 было зафиксировано «модульный меш — только у игрока; враги/NPC — единым склеенным мешем (Android: меньше draw call)». Ринат (2026-06-12) пересмотрел: модульность изначально проектировалась **на всех** персонажей — это объясняет, почему `Head/Torso/Legs` лежат в мастер-базе `AMasterHumanoidCharacter`, а не в `APlayerCharacter`.

**Решение (Ринат, 2026-06-12) — РАЗВОРОТ решения Фазы 0.** **Все персонажи (враги и дружественные NPC) — модульные** (Head/Torso/Legs из общей библиотеки частей на общем скелете). Спаунер врагов/NPC собирает персонажа из случайных/заданных модулей → дёшево в производстве + много визуальных вариаций из малого набора ассетов.

**Рычаг стоимости на Android — runtime skeletal mesh merge (склейка при спавне).**
- Авторинг/хранение — модульное (библиотека частей).
- Спаунер выбирает части → собирает → **при спавне склеивает в ОДИН скелетный меш** (UE Skeletal Merging: `USkeletalMergingLibrary` / `FSkeletalMeshMerge`) → **1 draw call** на персонажа, как у «единого меша», но собранного из модулей.
- **Враг/NPC** (снаряжение в рантайме не меняют) → merge один раз при спавне.
- **Игрок** (меняет броню вживую) → живые модульные компоненты, либо переклейка (re-merge) при смене экипировки.
- Итог: вариативность + дешёвое производство И стоимость одного меша на Android.

**Калибровка уверенности.**
- Модульность-для-всех + спаунер-сборка как практика — выведено/проверено (стандартный приём; совпадает с исходным замыслом Рината).
- Наличие Skeletal Merging **в UE 5.5** — НЕ подтверждено первоисточником на момент решения; **обязательна живая проверка cpp-dev/unreal-operator** (плагин Skeletal Merging) перед закладкой в код.
- **Фолбэк (если merge в 5.5 неудобен/недоступен):** живые модульные компоненты + Leader Pose Component (одна анимация) + LOD + разумный лимит одновременных врагов. Для MVP (мало врагов) приемлемо без merge.

**Влияние на архитектуру.** Отменяет рекомендацию «опустить модульность вниз в `APlayerCharacter`» (см. tech-design — снято). `AEnemyCharacter` наследует `AMasterHumanoidCharacter` **как есть** (модульные меши остаются в базе). Merge — отдельный слой/утилита, подключается как оптимизация (ориентир — когда врагов станет много, ~Фаза 3+); в Фазе 1 (1 бандит) — модульные компоненты без merge, не оптимизируем преждевременно.

**Самоограничения (чтобы держать «достаточно просто»):** ровно 3 слота (Head/Torso/Legs); смена = подмена целого меша слота; один общий скелет для всех вариантов всех слотов (дисциплина modeler-3d); части моделить внахлёст (без щелей на стыках).

**Статус: ПРИНЯТО (Ринат, 2026-06-12).** Дополняет GDD ч.2/ч.4/ч.7.4. Skeletal Merging в 5.5 — на живой проверке перед использованием.
