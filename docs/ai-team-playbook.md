# AI-Team Playbook — переиспользуемый опыт для создания новых ИИ-команд

> **Назначение.** Project-agnostic копилка проверенного опыта, паттернов и уникальных находок, полученных при работе команды (game-lead + специалисты + QA-пилот «Сборщик»). Ведут СОВМЕСТНО: **game-lead** (оркестрация/разработка) и **«Сборщик»** (QA/тестирование на Computer Use). Цель — из этого позже собрать **промпт-инструкцию + файлы контекста + ТЗ для НОВОЙ ИИ-команды** под другую задачу.
> **Правило ведения:** сюда — только ОБОБЩАЕМОЕ (паттерны, что сработало/провалилось, почему). Проектную специфику — в `context/*` и `docs/decisions.md` (ADR). Дополнять по мере находок.
> Версия 0.1 · начат 2026-06-13 (проект ContrarySurvivor, UE5.5).

---

## 1. АРХИТЕКТУРА КОМАНДЫ
- **Плоское дерево, глубина 1:** оркестратор (`game-lead`) → специалисты-терминалы. **Субагент НЕ порождает субагентов** (ограничение платформы) → если нужен другой спец, специалист возвращает результат лиду, лид делегирует дальше.
- **Потолок параллелизма ≤3** одновременно (чтобы не раздувать контекст/ресурсы).
- **Роли тул-скоупятся** (frontmatter `tools:`) — спец видит только нужные инструменты (напр. модельер не пушит в master). Это фича, не «урезанность».
- **Имена напарников = по роли** (cpp/unreal/modeler/qa/concept) — читаемость трейса.
- **Project-agnostic ядро + проект как загружаемый контекст** (диздок/ТЗ грузится в лида, не висит постоянно).
- **frontmatter (`tools:`, `mcpServers:`) кешируется на старте сессии** → правки определений агентов вступают в силу ТОЛЬКО после рестарта сессии. Закладывать заранее.

## 2. ПРОЦЕСС РАЗРАБОТКИ
- **Источник истины — диздок/ТЗ.** Чего нет — не выдумывать, эскалировать владельцу.
- **Фазы = играбельные инкременты с пруфом.** Роудмап утверждается ОДИН раз, дальше — рабочий ориентир. Вертикальный срез сначала.
- **Цикл фазы:** строим в фича-ветке → мерж-гейт (qa: реальная сборка + артефакт) → merge `--no-ff` → отчёт. Каждый шаг — с доказательством.
- **Мерж-гейт технический:** маркер-файл (`QA_OK`)/хуки, создаётся лидом ТОЛЬКО после «добро» qa с артефактом, снимается сразу после push. Не коммитится.

## 3. 🔑 АВТОНОМНАЯ QA-ПЕТЛЯ С Computer-Use ТЕСТЕРОМ («Сборщик»)
**Находка дня:** тестера-человека в цикле можно заменить ИИ-агентом с Computer Use (реально играет/кликает/смотрит) — до оговорённой вехи; владелец ревьюит на вехе.
- **Петля:** лид собирает (ресурс ЗАКРЫТ) → сигнал «🟢 ГОТОВО К ТЕСТУ + чеклист + состояние ресурса» → тестер прогоняет → отчёт + «🔵 свободно» → лид чинит → повтор до приёмки.
- **Полномочия тестера** настраиваются (у нас: решает до MVP, эскалирует крупное; на вехе — владелец).
- **Тестер честен:** PASS только по факту; чего не смог — пишет честно; «ощущение»/тонкое — оставляет владельцу.

## 4. 🔑 QA-ХАРНЕСС (ЭНЕЙБЛЕР для Computer-Use тестирования)
**Computer Use НЕ МОЖЕТ надёжно:** открыть игровую консоль (раскладка/фокус), кликать UI (захват мыши в play-режиме), рассмотреть мелкую 3D-деталь (камера сверху). Медленный, слаб в реакционном геймплее.
**Решение (строить ЗАРАНЕЕ под механики каждой фазы):**
- тест-действия на ОБЫЧНЫЕ КЛАВИШИ (не консоль): выдать предметы, экип, сделка, телепорт к NPC, зачесть прогресс и т.п.;
- структурированные ЛОГ-строки (отдельная категория, префикс напр. `QA:`) на ключевые события → тестер верифицирует ФУНКЦИЮ по логу, даже не видя UI;
- debug-камера/телепорт-клавиши для осмотра;
- подавить экранный спам.
→ Тестер закрывает ~80% (функция/визуал-статика), «фид»/мышь/баланс — финальный проход владельца. **Играем по сильным сторонам каждого.**

## 5. 🔑 MCP-НЕЗАВИСИМОСТЬ / HEADLESS-КОНВЕЙЕРЫ
- **Живые MCP-каналы к тяжёлым приложениям ФЛАККИ:** падают (троттлинг/тайминг), теряют инжект тулзов в свежие субагенты, требуют ручного reconnect владельцем (вне инструментов агента).
- **Надёжный обход — headless-CLI с исполнением скрипта:** напр. UE `UnrealEditor-Cmd "<proj>" -ExecutePythonScript="<abs.py>" -unattended -nopause -nosplash` (вывод в лог-файлы, НЕ stdout!); Blender `blender -b --factory-startup --python <скрипт>`. Без окна/сокета/MCP — только CLI+скрипт.
- **Приоритет инструментов:** (1) нативный коннектор → (2) встроенное → (3) MCP (если канал жив) → (4) **headless-коммандлет (когда MCP флакки)** → (5) Computer Use (крайний случай).
- **Граница headless:** годен для детерминированных операций (импорт/правки ассетов/материалы/сборка). НЕ годен для редактирования визуальных графов (UI/anim-blueprint) и для «поиграть» — это Computer Use / редактор.

## 6. КОММУНИКАЦИЯ / АСИНХРОН / ОЧЕРЁДНОСТЬ РЕСУРСА
- **Каналы = файлы в git** (владелец читает/пишет с телефона через веб-GitHub): лид→все (STATUS), тестер→лид (TEST_REPORT), владелец→лид (INBOX). Раздельные файлы → нет конфликтов; перед своим push — `pull`.
- **Фоновые watchers** (цикл `git fetch` + сравнение хеша/блоба, выход при изменении → платформа будит агента) — асинхронное «входящее». ⚠️ после рестарта сессии — поднять заново.
- **Очерёдность общего ресурса (редактор/устройство) — строго по сигналам + lock-файл** (тестер кладёт `*_TESTING.lock` на время; никогда одновременно). Не закрывать тяжёлое приложение без нужды (экономит минуты на перезапуск) — указывать состояние в сигнале.

## 7. АНТИ-ГАЛЛЮЦИНАЦИЯ (обязательно для всех ролей)
- **Пруф, а не утверждение:** «собралось/тест прошёл» — только с реальным логом; «закоммичено» — с `git`. Фабриковать запрещено (включая ВРЕМЯ — брать системное `date`, не выдумывать).
- **API/синтаксис — из доков конкретной версии**, не по памяти; перед использованием подтвердить наличие.
- **Состояние — читать реальные файлы/дерево**, не действовать по памяти.
- **Дизайн-решения — эскалировать**, не гадать. Допущения озвучивать явно. Калибровка: проверено / выведено / предположение.

## 8. ОБЩИЕ ИНЖЕНЕРНЫЕ ГРАБЛИ (обобщаемые)
- Классификатор автономии блокирует само-модификацию конфига (расширение прав/правка startup) без ЯВНОЙ авторизации владельца в сообщении — закладывать прямую фразу-разрешение.
- Тяжёлый редактор держит СТАРЫЙ бинарь в памяти → после пересборки кода нужен перезапуск; сборка кода требует ЗАКРЫТОГО редактора (иначе линковка-лок).
- Перед сдачей фазы — САМОТЕСТ силами команды (тестер/скрины), починить, ПОТОМ владельцу: его время = подтвердить, не искать баги.
- Большой непроверенный стек копить нельзя — не строить фазу N+1 поверх непринятой фазы N сверх разумного.

## 9. ДЛЯ БУДУЩЕЙ ЗАДАЧИ «СОЗДАТЬ НОВУЮ КОМАНДУ»
Артефакт (делаем СОВМЕСТНО с «Сборщиком»): **промпт-инструкция оркестратора + промпты ролей + файлы контекста (`context/*`, ADR, этот плейбук) + ТЗ под задачу.** game-lead даёт оркестрацию/процесс/анти-галлюцинацию/tooling; «Сборщик» — QA-петлю/харнесс/Computer-Use практику. Адаптировать роли под домен новой задачи (не обязательно геймдев), но каркас (разделы 1-8) переносится.
